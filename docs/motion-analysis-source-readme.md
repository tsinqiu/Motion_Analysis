# Motion Analysis

这是一个面向数据库课程期末大作业的运动数据分析项目。当前阶段的重点是：**解析手表导出的 FIT 文件，并尽可能完整地写入 MySQL 数据库**。

后续可以继续扩展云端服务、前端页面、App 展示、运动趋势、健康度评估、运动日历统计等功能；但这些暂时不是当前实现重点。当前先把数据底座做好，也就是把 FIT 文件中的各种运动数据稳定、完整、可重复地导入数据库。

## 当前目标

项目当前优先完成以下内容：

- 读取 `Data` 目录中的 FIT 文件。
- 解析 FIT 文件中的活动摘要、逐点记录、分段数据和运动事件。
- 将解析结果结构化保存到 MySQL。
- 保留原始 FIT 文件信息，方便后续追溯和重复解析。
- 提供基础 SQL 查询，用于检查导入结果和进行简单分析。

换句话说，当前阶段先解决“数据从 FIT 文件进入数据库”的问题。只要数据库中的数据结构设计清楚，后续做图表、趋势、日历、云端访问都会更容易。

## 数据来源

`Data` 目录用于存放手表导出的 FIT 运动文件。当前目录里只是先放入了几类典型运动样例，方便开发和验证；后续可以继续加入更多 FIT 文件，系统设计不应限制活动数量。

正式实现时只把 `.fit` 文件作为核心数据源。目录中的 `.gpx`、`.tcx`、JSON 文本和纯文本报告可以用来理解数据含义、核对解析结果，但不作为系统主流程依赖。

FIT 文件中通常包含：

- 活动摘要：运动类型、开始时间、距离、时长、平均心率、最大心率、平均速度、热量等。
- 逐点记录：每个采样点的时间、经纬度、海拔、距离、速度、心率、步频、功率等。
- 分段信息：每圈或每段的距离、用时、心率、速度、功率等。
- 运动事件：开始、暂停、恢复、结束等。
- 设备和文件信息：设备、创建时间、文件类型等。

## 数据库设计方向

数据库使用 MySQL。建议围绕 FIT 文件解析结果建立数据表，而不是只保存一个文件或一段文本。

建议包含以下数据表：

- 原始文件表：记录 FIT 文件名、大小、哈希值、导入时间和保存路径。
- 活动主表：记录一次运动的基本信息，例如运动类型、开始时间、活动名称等。
- 活动摘要表：记录总距离、总时长、平均心率、最大心率、平均速度、热量等。
- 轨迹点表：记录每个采样点的时间、位置、速度、心率、功率等。
- 分段表：记录每圈或每段的统计结果。
- 事件表：记录开始、暂停、恢复、结束等运动事件。
- 扩展指标表：保存心率区间、功率区间、训练负荷等不一定每个文件都有的指标。

其中轨迹点表最重要，因为它保存的是“每时每刻”的运动状态，后续地图轨迹、心率曲线、速度曲线、功率曲线都依赖这张表。

## 当前阶段应完成的查询

为了证明数据库导入有效，当前阶段建议至少准备这些查询：

- 查询已导入的 FIT 文件列表。
- 查询所有运动活动的基本信息。
- 查询单次活动的摘要数据。
- 查询单次活动的全部轨迹点。
- 查询单次活动的心率-时间数据。
- 查询单次活动的速度-时间数据。
- 查询单次活动的分段数据。
- 统计不同运动类型的活动数量、总时长、总距离和平均心率。

这些查询已经足够体现数据库表结构、主外键关系、索引和基础分析能力。

## 执行效率考虑

运动数据的逐点记录数量会比较大。一次长距离跑步就可能产生几千个采样点，后续文件增多后，轨迹点表会成为主要数据量来源。因此当前设计需要兼顾导入速度和查询效率。

本项目采用的策略：

- 常用字段单独存成数据库列，例如时间、经纬度、距离、速度、心率、功率。
- 默认不把每条 FIT 原始消息 JSON 全量写入数据库，避免数据库和导入 SQL 过大。
- 轨迹点表建立活动 ID + 时间、活动 ID + 距离等索引，方便查询曲线和轨迹。
- 通过文件哈希识别重复 FIT 文件，避免重复导入。
- 如果后续需要调试或追溯完整 FIT 消息，可以再开启完整消息导入模式。

当前优先保证结构化数据查询快、导入稳定；完整原始文件仍保留在文件目录中，不需要把所有原始内容都重复塞进数据库。

## 后续扩展方向

当前重点不是一次性完成所有展示和云端功能，但可以在报告中说明后续扩展方向：

- 文件上传：通过页面或接口上传新的 FIT 文件。
- 前端展示：活动列表、活动详情、地图轨迹、心率/速度/功率曲线。
- 云端服务：将后端和数据库部署到云服务器，支持随时访问。
- 运动趋势：按周、月统计运动次数、距离、时长、心率变化。
- 健康度分析：根据心率、训练负荷、恢复情况等做简单评估。
- 运动日历：按日期展示训练记录。
- 多运动类型对比：比较跑步、骑行、力量训练等不同活动。

这些功能都建立在当前数据库导入工作之上。因此当前优先级是先把 FIT 数据解析完整、入库稳定。

## 推荐实现顺序

1. 了解 FIT 文件中常见消息和字段。
2. 编写或引入 FIT 解析器。
3. 设计 MySQL 表结构。
4. 将 FIT 的活动摘要、轨迹点、分段、事件写入数据库。
5. 处理重复导入问题，例如通过文件哈希避免重复记录。
6. 编写基础查询 SQL。
7. 用样例报告核对距离、时长、心率等关键结果。
8. 在数据入库稳定后，再考虑上传接口、前端展示和云端部署。

## 作业亮点

这个项目适合数据库期末大作业，因为它包含：

- 真实运动数据。
- FIT 二进制文件解析。
- MySQL 结构化建模。
- 活动、轨迹点、分段、事件等多表关系。
- 时间序列数据和空间轨迹数据。
- 可扩展到前端展示、云端服务和健康分析。
